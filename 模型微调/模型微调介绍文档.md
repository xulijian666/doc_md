# 模型微调：训练保险领域的需求分析模型

---

## 一、我们要解决什么问题？

### 场景描述

在日常工作中，我们会收到大量的**寿险需求规格说明书**（简称"需规"）。拿到一份需规之后，业务人员需要做一系列分析工作：

- 这份需求涉及哪些保险实体？（投保人、被保人、受益人、险种、给付责任……）
- 这些实体之间是什么关系？（投保人购买了什么险种、险种包含哪些给付责任……）
- 这些内容和我们已有的知识体系怎么对得上？

换句话说，就是把一份"自然语言写的需求文档"，**翻译成**一套"结构化的、标准化的分析结果"。

### 现状痛点

| 痛点 | 具体表现 |
|------|----------|
| **慢** | 一份复杂的寿险需规，人工分析可能需要几天 |
| **容易漏** | 人工阅读长文本，难免遗漏某些条款细节 |
| **标准不统一** | 不同人分析同一份需规，输出结果的格式和深度可能完全不一样 |

### 我们的目标

> 让 AI 自动完成这个"翻译"工作——输入一份寿险需规，输出标准化的结构化分析结果。

---

## 二、为什么通用 AI 做不了这件事？

### 一个类比

假设你招了一个刚毕业的高材生：

- 他**智商很高**，语言能力一流，什么文章都能读懂
- 但他**从没接触过保险行业**，不知道什么是"现金价值"、"复效期"、"减额交清"
- 他更不知道我们内部有一套叫**"本体"**的知识规范体系

你直接把一份寿险需规丢给他，让他按规范输出分析结果——他只会用大白话跟你聊，甚至会一本正经地胡编乱造。

### 通用 AI 的问题

| 能力 | 通用 AI 的表现 |
|------|----------------|
| 语言理解 | 很强，读得懂中文 |
| 保险专业知识 | 很弱，不懂行业术语和业务逻辑 |
| 输出格式规范 | 不稳定，即使你反复强调"请输出标准格式"，它依然会跑偏 |

### 核心矛盾

> **它有智商，但没有行业手艺。**

所以我们需要一种方法，教它学会这门手艺。这个方法就是——**微调**。

---

## 三、那怎么教它？——微调是什么？

### 核心概念

**微调（Fine-Tuning）**，简单说就是：在一个已经训练好的通用 AI 模型基础上，用特定的"教材"再训练一遍，让它学会某种特定的行为方式。

### 一个类比：入职培训

还是那个高材生的比喻：

| 阶段 | 做什么 | 说明 |
|------|--------|------|
| 大学教育 | 预训练 | AI 模型出厂前的"通用教育"，让它学会语言和常识 |
| **入职培训** | **微调** | 拿出《优秀员工工作案例集》，手把手教它怎么干活 |

通过几千次的规范训练，这个高材生的工作习惯、思维链条、输出格式被扭转成一个符合业务规范的"专才"。

### 关键澄清

> **微调不是让 AI 去"背诵"保险条款，而是教它"做事的方法"。**

| 错误理解 | 正确理解 |
|----------|----------|
| 把所有寿险条款塞进 AI 脑子里 | 教 AI 一套"看到需求→怎么拆解→怎么输出"的思维范式 |
| 让它记住每一份文档 | 让它学会按规范格式处理任何新文档 |

打个比方：你不需要厨师背下整本菜谱，你只需要教会他"切菜、控火、调味"的手艺，他就能应对各种新菜式。

## 四、微调有哪三种主流方向？

目前业界主流的微调方式有三种，它们各司其职，解决不同层面的问题。

### 4.1 CPT —— 继续预训练（让它多读专业书）

**类比：让大学生去读保险专业的研究生。**

| 维度 | 说明 |
|------|------|
| **干什么** | 让 AI 大量阅读寿险条款、保险法规、精算术语等专业文本 |
| **数据形式** | 纯文本，不需要人工编写问答（直接把文档丢给它读） |
| **解决什么问题** | 扩充行业常识——让它从"外行"变成"对保险有基本认知" |
| **比喻** | 大学生去读研，补专业背景知识 |
| **适用场景** | 当基座模型对行业术语理解不足时，推荐先用 CPT 补充专业知识 |

**训练数据集长什么样？**

CPT 的数据最简单——就是**纯文本**，不需要任何人标注。直接把专业文档丢给 AI 读就行。

| 数据来源 | 示例 |
|----------|------|
| FIBO 本体文档 | "Financial Industry Business Ontology (FIBO) defines InsurancePolicy as a contract between an insurer and policyholder..." |
| 寿险条款原文 | "定期寿险的观察期为180天，观察期内因疾病身故仅退还已交保费，观察期后身故按基本保额给付身故保险金..." |
| 保险法规 | "根据《保险法》第十六条，投保人故意不履行如实告知义务的，保险人对于合同解除前发生的保险事故，不承担赔偿或者给付保险金的责任..." |
| 公司内部需规 | "需求编号：REQ-LIFE-2024-0032。功能描述：新增保单贷款功能，贷款金额上限为保单现金价值的80%..." |

> **类比：就像给学生塞了一摞专业课本和公司内部资料，让他自己读，不用出考题。**

### 4.2 SFT —— 有监督微调（手把手教它干活）

**类比：带新人做项目，给标准答案照着学。**

| 维度 | 说明 |
|------|------|
| **干什么** | 用"问题→标准答案"的问答对，教会 AI 怎么处理需求文档 |
| **数据形式** | 一问一答，需要人工精心编写（比如：输入一段需规，输出标准分析结果） |
| **解决什么问题** | 学会"听指令、按规矩干活"——从会说大白话变成能输出规范格式 |
| **比喻** | 带着新人做项目，做一步教一步，手把手示范 |
| **适用场景** | 业界使用率最高的微调方式，适合需要教会 AI 特定行为模式和输出格式的场景 |

**训练数据集长什么样？**

SFT 的数据是**一问一答**，每一组都包含"输入"和"标准输出"。为了让 AI 学会思考过程，我们还会在中间加上"解题步骤"（这叫思维链，CoT）。

**一组完整的训练数据包含三个部分：**

```
【题目】（Instruction + Input）
"请分析以下寿险需规文本，提取核心实体，映射到本体类中，以标准 Markdown 格式输出。"

需求文本：定期寿险产品要求投保人在投保时如实告知健康状况，观察期为180天。
观察期内因疾病身故仅退还已交保费；观察期后身故按基本保额给付身故保险金。
保单犹豫期为15天，犹豫期内可全额退保。

【解题步骤】（Thought）
第一步：识别核心实体——"投保人"、"定期寿险"、"观察期"、"身故保险金"、"犹豫期"
第二步：在本体中找到对应类——投保人→FIBO:InsuredParty，定期寿险→FIBO:TermLifeInsurance
第三步：梳理关系——投保人 与 定期寿险 之间是 hasPolicy 关系
第四步：整理属性——观察期=180天，犹豫期=15天

【标准输出】（Response）
## [[定期寿险]]
- **本体类**：FIBO:TermLifeInsurance
- **投保人**：[[投保人]]（FIBO:InsuredParty）
- **观察期**：180天
- **犹豫期**：15天
- **给付责任**：[[身故保险金]]（FIBO:InsuranceBenefit）
  - 观察期内：退还已交保费
  - 观察期后：按基本保额给付
```

> **类比：就像老师批改作业——先看题目，再看解题过程，最后对标准答案。每一步都清清楚楚。**

### 4.3 DPO —— 偏好优化（纠偏、提升质量）

**类比：绩效考核，告诉它哪些回答好、哪些回答差。**

| 维度 | 说明 |
|------|------|
| **干什么** | 给 AI 看"好的回答"和"差的回答"，让它学会区分好坏 |
| **数据形式** | 好回答 vs 坏回答的对照组 |
| **解决什么问题** | 减少胡说八道（幻觉），让输出更靠谱、更符合预期 |
| **比喻** | 绩效考核——干得好发奖金，干得差扣绩效 |
| **适用场景** | 适合对输出质量有极高要求的场景，如降低幻觉、调整语气、确保安全合规 |

**训练数据集长什么样？**

DPO 的数据是**三元组**——同一个问题，给一个好回答和一个坏回答，让 AI 自己对比学习。

```
【同一个问题】
"请分析以下定期寿险需求，映射到本体类。"
（需求文本省略...）

【好回答】（Positive —— 模型应该学的）
## [[定期寿险]]
- **本体类**：FIBO:TermLifeInsurance
- **观察期**：180天
- **给付责任**：[[身故保险金]]（FIBO:InsuranceBenefit）

【坏回答】（Negative —— 模型应该避免的）
"定期寿险是一种人寿保险产品，它主要提供身故保障。在这份需求中，
我们可以看到有观察期和犹豫期的设定。观察期是180天，意味着在投保后的
180天内如果发生身故，保险公司只会退还保费。犹豫期是15天..."
（问题：用了大白话聊天，没有按规范格式输出，缺少本体映射）
```

> **类比：就像经理拿出两份报告做对比——"这份写得好，格式标准、分析到位；那份写得差，啰嗦没重点、格式混乱。以后照好的来。"**

**DPO 里"坏回答"通常有这些毛病：**

| 坏回答的典型问题 | 说明 |
|------------------|------|
| 用大白话聊天 | 没有输出规范的 Markdown 格式 |
| 本体映射错误 | 把"投保人"映射到了错误的类上 |
| 遗漏实体 | 需求里明明有"犹豫期"，输出里却没提 |
| 一本正经胡说八道 | 编造了需求文本里没有的内容（幻觉） |

### 4.4 小结：三者的关系

用一个完整的类比串起来：

```
CPT（读研）     →    SFT（实习）    →    DPO（绩效考核）
补充行业常识         学会做事方法          纠偏提升质量
```

**三种范式的关系：**

| 范式 | 数据准备难度 | 典型数据量 | 推荐搭配 |
|------|-------------|-----------|----------|
| CPT | 低（纯文本，无需标注） | 几 GB 专业文档 | 适合行业知识补充不足时使用 |
| SFT | 中（需编写问答对） | 2000~3000 条 | 适合需要教会 AI 特定行为模式时使用 |
| DPO | 高（需编写好/坏对照组） | 1000~2000 条 | 适合对输出质量有极高要求时使用 |

> **业界常见实践：** 大多数垂直领域场景下，SFT 是基础标配；对于金融、保险等专业知识密集的领域，CPT+SFT 的组合更为常见；对输出质量和安全性要求极高的产品，会在 SFT 基础上增加 DPO 环节。

## 五、结合寿险需求分析：我们的方案是什么？

### 5.1 我们的知识底座：本体 + Markdown 知识库

在讲微调怎么用之前，先了解我们的知识架构——整个系统有两大知识底座：

| 组件 | 角色 | 类比 | 说明 |
|------|------|------|------|
| **Protégé 本体（TBox）** | 规定规则 | 法律 | 定义了保险世界的"类"（如投保人、险种、给付责任）和"关系"（如投保人购买险种），是绝对的规范 |
| **Markdown 知识库（ABox）** | 存放内容 | 百科全书 | 存放着海量的寿险条款、需求文档实例，用双链（`[[重疾险]]`、`[[等待期]]`）连成一张知识网络 |

它们的关系：

> **本体是"宪法"，规定了世界怎么分类；知识库是"图书馆"，按宪法的分类存放具体知识。**

### 5.2 微调在这里扮演什么角色？

有了本体和知识库，还需要微调干什么？

答案是：**需要一个"操作员"来连接它们。**

类比说明：

> 你有一本法律大全（本体）和一个巨大的图书馆（知识库），但你需要一个**专业的图书管理员**——他知道怎么根据问题去图书馆里查资料，查完之后怎么按照法律的分类标准整理出来。

**微调，就是培养这个"图书管理员"。**

具体来说，微调教会 AI 三样本事：

| 本事 | 说明 | 类比 |
|------|------|------|
| **本体感知** | 读到需求时，本能地找到对应的本体类（如 `FIBO:InsuredParty`）和属性（如 `hasPremium`） | 图书管理员看到书名就知道该放到哪个分类架上 |
| **知识关联** | 熟练使用 Markdown 双链语法，把分析结果和已有知识节点关联起来 | 知道这本书和那本书之间有引用关系，主动建立索引 |
| **格式纪律** | 输出永远是标准的、可直接被程序解析的 Markdown 格式 | 填表永远按模板来，绝不乱写 |

### 5.3 完整工作流程

把整个流程串起来：

```
┌──────────────────────┐
│  寿险需求规格说明书    │  ← 输入：一份自然语言写的需求文档
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│  微调后的 AI 模型      │
│                      │
│  ① 阅读理解需求文本   │
│  ② 匹配本体类和属性   │
│  ③ 关联知识库中的节点 │
│  ④ 输出标准格式结果   │
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│  结构化分析结果        │  ← 输出：标准化的 Markdown，可直接织入知识库
│  （带双链引用、        │
│    符合本体规范）      │
└──────────────────────┘
```

**类比：就像给工厂装了一条自动化生产线——原料（需求文档）进来，标准产品（结构化分析结果）出去。**

## 六、具体怎么落地？

围绕竞赛总纲的任务 C，我们分成两步走：

### 6.1 任务 C-a：训练数据集准备

**类比：根据不同的"培训方式"，编写不同类型的教材。**

前面介绍了三种微调范式，每种范式需要的数据格式不同。下面分别说明。

---

#### 6.1.1 CPT 数据集——"专业课本"

**类比：给学生准备一摞专业课本和参考资料，让他自己读。**

| 维度 | 说明 |
|------|------|
| **数据格式** | 纯文本，无需人工标注 |
| **数据来源** | FIBO 本体文档、寿险条款原文、保险法规、公司内部需规 |
| **典型数据量** | 几 GB 的专业文本 |
| **制作难度** | 最低——直接收集现有文档即可 |

**数据示例：**

```
【FIBO 本体文档片段】
Financial Industry Business Ontology (FIBO) defines InsurancePolicy as a
contract between an insurer and a policyholder, specifying the terms and
conditions of coverage...

【寿险条款原文】
定期寿险的观察期为180天，观察期内因疾病身故仅退还已交保费，
观察期后身故按基本保额给付身故保险金...

【公司内部需规】
需求编号：REQ-LIFE-2024-0032。功能描述：新增保单贷款功能，
贷款金额上限为保单现金价值的80%...
```

---

#### 6.1.2 SFT 数据集——"真题及详解"

**类比：编写一本《岗位操作手册及真题详解》，每道题都有标准答案和解题步骤。**

| 维度 | 说明 |
|------|------|
| **数据格式** | 思维链问答对（题目 → 解题步骤 → 标准答案） |
| **数据来源** | 专家编写 + AI 扩增 + 自动校验 + 专家精修 |
| **典型数据量** | 2000~3000 条 |
| **制作难度** | 中等——需要精心设计问答对 |

**一组完整的 SFT 数据包含三个部分：**

| 部分 | 内容 | 类比 |
|------|------|------|
| **题目** | 一段寿险需求文本 + 指令 | 考试卷上的题目 |
| **解题步骤** | AI 应该怎么一步步思考（思维链） | 标准答案的"解题过程" |
| **标准答案** | 符合规范的结构化 Markdown 输出 | 最终得分点 |

**三类题型均衡配置：**

| 题型 | 内容 | 目的 | 建议占比 |
|------|------|------|----------|
| **题型 A** | 需求文本 → 本体映射 | 学会识别保险实体，对到本体类上 | ~30% |
| **题型 B** | 需求文本 → Markdown 知识网络 | 学会用双链把分析结果织入知识库 | ~30% |
| **题型 C** | 需求场景 → 结构化评估 | 学会做功能拆解、冲突检查、工作量预估 | ~30% |

**SFT 数据生产流程：**

| 步骤 | 做什么 | 数量 |
|------|--------|------|
| 第一步 | 专家亲自编写"教科书级"完美范例 | ~50 条种子数据 |
| 第二步 | 用强大的 AI 模型学习范例，批量仿写 | 扩增至 2000~3000 条 |
| 第三步 | 程序自动检查格式合法性（括号配对、类名是否存在等），不合格的淘汰 | 自动过滤 |
| 第四步 | 专家抽样检查，修正业务逻辑错误 | 精修定稿 |

> **关键原则：1000 条高质量数据，远好于 10000 条粗制滥造的数据。质量永远比数量重要。**

---

#### 6.1.3 DPO 数据集——"好报告 vs 坏报告"

**类比：经理拿出两份报告做对比，告诉员工哪份写得好、哪份写得差。**

| 维度 | 说明 |
|------|------|
| **数据格式** | 三元组（同一个问题 + 好回答 + 坏回答） |
| **数据来源** | 好回答来自 SFT 标准答案，坏回答由人工或 AI 刻意构造 |
| **典型数据量** | 1000~2000 条 |
| **制作难度** | 最高——需要同时编写好/坏两组回答，且"坏"得要有代表性 |

**一组完整的 DPO 数据包含三个部分：**

| 部分 | 内容 | 说明 |
|------|------|------|
| **问题** | 一段需求文本 | 三种范式共用的输入 |
| **好回答（Positive）** | 符合规范的结构化输出 | AI 应该学会的 |
| **坏回答（Negative）** | 有典型毛病的输出 | AI 应该避免的 |

**"坏回答"的典型毛病：**

| 问题类型 | 具体表现 |
|----------|----------|
| 用大白话聊天 | 没有输出规范的 Markdown 格式，而是写了一段散文 |
| 本体映射错误 | 把"投保人"映射到了错误的本体类上 |
| 遗漏实体 | 需求里明明有"犹豫期"，输出里却没提 |
| 胡说八道 | 编造了需求文本里没有的内容（幻觉） |

---

#### 6.1.4 三种数据集对比总结

| 维度 | CPT 数据 | SFT 数据 | DPO 数据 |
|------|----------|----------|----------|
| **类比** | 专业课本 | 真题及详解 | 好报告 vs 坏报告 |
| **格式** | 纯文本 | 思维链问答对 | 好/坏回答三元组 |
| **需要标注** | 否 | 是 | 是 |
| **制作难度** | 低 | 中 | 高 |
| **典型数据量** | 几 GB | 2000~3000 条 | 1000~2000 条 |

### 6.2 任务 C-b：小基座模型微调

**类比：培养一个"打下手的实习生"，帮教授（大模型）准备材料、做预处理。**

#### 什么是"小基座模型"？

| 类型 | 参数量 | 类比 | 代表 |
|------|--------|------|------|
| **大模型** | 700亿+ | 全能教授，理解力强，但身价高、速度慢 | GPT-4、Claude |
| **小基座模型** | **70~140亿** | **精干的实习生，聪明、便宜、手脚麻利** | **Qwen2.5-7B、Llama-3.1-8B** |

#### 小基座模型在系统中的角色——"助手"而非"主角"

在实际的需求分析流程中，小基座不是最终输出分析结果的那个"专家"，而是**帮大模型打下手的助手**。具体承担三类工作：

| 角色 | 做什么 | 类比 |
|------|--------|------|
| **数据标注助手** | 把原始需规文本初步标注成结构化数据，给大模型微调准备教材 | 帮老师批改选择题的课代表 |
| **数据扩增助手** | 学习专家写的 50 条种子范例，批量仿写几千条训练数据 | 照着样卷誊写练习题的学委 |
| **预处理工具** | 对需求文档做初步的实体识别和文本分类，减轻大模型负担 | 公司前台，先初筛访客，只让重要的进去见老板 |

#### 完整的工作流：小基座 + 大模型协同

```
┌──────────────────────────────────────────────────┐
│                                                  │
│   寿险需求规格说明书                               │
│          │                                       │
│          ▼                                       │
│   ┌────────────────────────────┐                │
│   │  小基座模型（助手）          │                │
│   │                            │                │
│   │  ① 初步实体识别             │                │
│   │  ② 文本预处理/拆分          │                │
│   │  ③ 标注训练数据             │                │
│   └─────────────┬──────────────┘                │
│                 │                                │
│                 ▼                                │
│   ┌────────────────────────────┐                │
│   │  大模型（最终专家）          │                │
│   │                            │                │
│   │  ① 接收预处理结果           │                │
│   │  ② 结合本体做深度映射       │                │
│   │  ③ 结合知识库做推理         │                │
│   │  ④ 输出最终结构化分析       │                │
│   └────────────────────────────┘                │
│                                                  │
└──────────────────────────────────────────────────┘
```

#### 为什么选小模型来做助手？

| 优势 | 说明 | 类比 |
|------|------|------|
| **省钱** | 只需要一张普通显卡就能运行，不需要昂贵的服务器集群 | 实习生工资低，养得起 |
| **安全** | 可以完全在内部网络运行，数据不出门 | 在自己公司培训，不用担心商业机密外泄 |
| **快** | 推理速度快，批量处理数据时效率极高 | 手脚麻利，干杂活效率高 |
| **可微调** | 体量小，微调成本低，可以快速定制专属的数据标注和预处理能力 | 实习生可塑性强，教什么学什么 |

#### 怎么微调小基座？

| 方法 | 做什么 | 什么时候用 |
|------|--------|------------|
| **SFT** | 用精心准备的教材教它学会数据标注和预处理的手艺 | 核心步骤 |
| **CPT** | 如果它连基础保险术语都不认识，先让它读大量专业书补补课 | 视情况而定 |

#### 微调后的效果

| 微调前 | 微调后 |
|--------|--------|
| 不认识保险术语和本体类 | 能初步识别需求文本中的保险实体 |
| 输出格式随心所欲 | 标注数据时格式标准、规范 |
| 需要长长提示词才能干活 | 简单指令就能高效完成数据预处理任务 |
| 批量处理效率低 | 可以快速处理大量需规，为大模型备好"半成品" |

## 七、总结

用一句话说清楚整个方案：

> **我们用大模型做最终的需求分析，同时用小基座模型做数据准备和预处理，两者协同工作。**

整个方案的核心逻辑：

```
本体（规矩） + 知识库（资料） + 小基座（助手预处理）+ 大模型（专家出结果）= 自动化需求分析
```

| 组件 | 角色 | 一句话 |
|------|------|--------|
| **Protégé 本体** | 规定规矩 | 告诉 AI 保险世界怎么分类、关系怎么定义 |
| **Markdown 知识库** | 提供资料 | 存放海量条款和需规，供 AI 随时查阅 |
| **训练数据集** | 培训教材 | 教小基座学会数据标注和预处理的手艺 |
| **小基座模型** | 打下手的助手 | 负责数据标注、扩增、预处理，帮大模型备好材料 |
| **大模型** | 最终专家 | 接收预处理结果，结合本体和知识库，输出最终需求分析 |
