# 小基座模型在寿险本体项目中的角色

## 一、什么是小基座模型？

### 定义

如果把 GPT-4 比作"全知全能的超级工业巨头"，小基座模型就是**"体型轻巧、五脏俱全、极具性价比的特种工业单兵"**。

### 技术特征：什么体量才算"小"？

| 类别 | 参数量 | 代表模型 |
|------|--------|----------|
| **小基座模型** | 7B~14B（B = Billion，十亿） | Qwen2.5-7B/14B、Llama-3.1-8B、Gemma-2-9B |
| 大模型 | 72B+ | Qwen-72B、Llama-3.1-70B |
| 闭源巨头 | 万亿级 | GPT-4、Claude |

### 为什么 7B~14B 是"黄金尺寸"？

```
低于 7B（1B/3B）
    → 中文理解、复杂逻辑推理、格式控制能力断崖式下跌
    → 很难听懂复杂的本体映射指令

7B~14B（黄金尺寸）
    → 刚好达到大模型能力的"涌现阈值"
    → 性价比最高

高于 72B+
    → 更聪明，但硬件消耗呈指数级上升
    → 中小企业和比赛团队负担不起
```

## 二、为什么寿险本体项目要选"小"基座？

### 理由一：算力经济学——一张显卡就能跑

| 模型 | 硬件需求 | 成本 |
|------|----------|------|
| 72B+ 大模型 | 几台 A800/H800 服务器协同 | 中小企业根本负担不起 |
| 7B 小基座 | 量化后只需一张消费级显卡（RTX 4090）或企业级单卡（A10） | 本地化私有部署、低成本微调成为可能 |

### 理由二：数据安全与本地化

- 寿险项目涉及内部核心资产、未公开需规文档、Protégé 本体
- 这些数据**绝对不能上传给公网闭源大模型**（如 GPT-4 API）
- 小基座模型可以**完全断网部署**在企业内部局域网服务器
- 保障数据不出境、不泄露

### 理由三：推理延迟与响应速度

- 下游应用（需求分析、工作量评估）需要频繁漫游 Markdown 知识库
- 大模型吐一个字一两秒，系统串联吞吐量极低
- 小基座模型推理速度快，带来丝般顺滑的自动化流转体验

## 三、小基座模型在寿险本体中的角色定位

### 辩证认知："不能"与"能"

| 维度 | 说明 | 比喻 |
|------|------|------|
| **不能** | 原生通用知识库贫瘠，不认识公司专用术语，不知道 Markdown 库里昨天更新了什么 | 失忆症患者 |
| **能** | 原生泛化语言能力、语法解析能力、逻辑直觉完全在线 | 绝顶聪明、手脚麻利的年轻学徒 |

### 核心定位：通用计算引擎 + 图谱导航员

```
小基座模型 = 一台高性能的、不带任何预设数据的"通用计算引擎"
```

## 四、小基座模型怎么微调？

### 微调目标

不是让模型**记住**保险知识，而是让它**学会手艺**：

| 手艺 | 说明 |
|------|------|
| 本体感知 | 读到需求时本能找 TBox 的 Class 和 Property |
| Markdown 操作 | 精通双链 `[[Page]]`、YAML Front Matter |
| 格式纪律 | 输出永远是干净、可解析的标准格式 |
| 翻译思维 | 像保险专家一样拆解文本、做本体映射 |

### 微调方式

| 阶段 | 方法 | 说明 |
|------|------|------|
| **可选：CPT** | 继续预训练 | 如果小基座连"现金价值""复效期"都不懂，先灌专业词汇 |
| **必选：SFT** | 有监督微调 | 用 CoT 思维链问答对教"本体映射手艺" |
| 暂不需要：DPO | 偏好优化 | 比赛阶段性价比低 |

### 微调数据

- **~50 条**种子数据（专家手工编写，教科书级完美范例）
- **2000~3000 条**扩增数据（LLM 自动生成 + 语法校验 + 专家精修）
- 三类题型均衡分布：
  - 题型 A：自然语言 → Protégé 本体映射（~30%）
  - 题型 B：文本解构 → Markdown 双链网络（~30%）
  - 题型 C：需求场景推演 → 结构化评估（~30%）

### 微调后的效果

| 微调前 | 微调后 |
|--------|--------|
| 用大白话长篇大论 | 起手就是标准 Markdown |
| 不认识 FIBO 本体 | 本能找 Class 和 Property |
| 双链格式随机 | 每个双链引用都精准 |
| Prompt 要写长长规范 | Prompt 极简，注意力全在需求理解 |

## 五、总结：小基座模型在寿险本体中的完整定位

| 问题 | 答案 |
|------|------|
| **是什么？** | 高性能通用计算引擎，不带任何预设数据 |
| **为什么选它？** | 体积小、跑得快、便宜、完全受控、数据不出境 |
| **拿来做什么？** | 图谱导航员——读 Protégé 铁律，漫游 Markdown 领土，输出严谨分析 |
| **怎么微调？** | SFT 必选（教手艺）+ CPT 可选（补常识），用 CoT 思维链数据 |
| **不负责什么？** | 不负责背诵具体保险知识（那是 RAG 的事） |

> 你不需要它自带任何保险知识。你选择它，是因为它体积小、跑得快、便宜且完全受你控制。你通过微调把它改造成"图谱导航员"，它负责在最前线干脏活累活——读取 Protégé 本体的铁律，漫游 Markdown 本领土，为你输出最严谨的需求分析产出。
