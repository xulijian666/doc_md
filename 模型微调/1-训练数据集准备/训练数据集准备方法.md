# 训练数据集准备方法

## 总体原则

微调不是为了让模型硬背寿险条款，而是教它学会**"本体映射的手艺"**和**"Markdown 语法编排的纪律"**。

训练集不能是杂乱的无标注文本，必须是高质量的、包含思考过程的**"行为示范集"**。

---

## 一、数据结构：采用"思维链（CoT）"问答对

### 为什么用 CoT？

为了让小基座模型学会复杂的本体映射，不能直接给输入→输出，而要教它**"解题步骤"**。

### 一条标准训练数据的三部分

| 部分 | 内容 | 示例 |
|------|------|------|
| **输入（Instruction + Input）** | 任务指令 + 具体寿险需求文本 | "请分析以下寿险需规文本，提取核心实体，并将其映射到 Protégé 本体类中，以标准 Markdown 格式输出。" + [寿险需求文本] |
| **思考过程（Thought/Reasoning）** | 模型的推导步骤 | "第一步，识别到'投保人'，对应 FIBO:InsuredParty 类... 第二步，识别到'身故给付'，属于 FIBO:InsuranceBenefit... 第三步，梳理关系，使用属性 hasBenefit..." |
| **输出（Response）** | 极其规整的 Markdown 文本 | 包含 [[双链]] 节点定义、符合本体规范的属性列表 |

---

## 二、核心"题型"设计：三类数据集

### 题型 A：自然语言 → Protégé 本体映射

| 维度 | 说明 |
|------|------|
| **目的** | 教会模型识别寿险文本，自动做 TBox 概念对齐 |
| **输入** | 寿险条款（如犹豫期退保规定） |
| **输出** | 准确找出对应的 Class 和 Property，并说明映射理由 |

### 题型 B：文本解构 → Markdown 双链网络生成

| 维度 | 说明 |
|------|------|
| **目的** | 训练模型成为"Markdown 管道工"，学会用双链把实例连接起来 |
| **输入** | 零散的需求规约文本 |
| **输出** | 标准 Markdown 块，自动带上 `[[相关险种]]`、`[[业务模块]]` 双链标签，可直接织入现有知识库 |

### 题型 C：需求场景推演 → 结构化评估（为 D 步骤铺垫）

| 维度 | 说明 |
|------|------|
| **目的** | 训练模型结合本体和知识库进行实际应用 |
| **输入** | 具体功能开发需求 |
| **输出** | 包含功能点拆解、基线冲突检查、工作量预估的 Markdown 表格 |

### 数据分布建议

| 题型 | 建议占比 | 说明 |
|------|----------|------|
| 题型 A（本体映射） | ~30% | 基础能力 |
| 题型 B（双链生成） | ~30% | 核心手艺 |
| 题型 C（场景推演） | ~30% | 应用能力 |
| 其他/混合 | ~10% | 综合题型 |

> 如果 90% 都是简单名词映射，模型永远学不会复杂的"工作量评估表格生成"。

## 三、数据生产路径：LLM 自动化 + 专家校验

### 四步流水线

```
┌──────────────────────────────┐
│ 1. 种子数据（人工编写）        │
│    ~50 条教科书级完美映射范例  │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│ 2. 黄金模型（LLM 扩增）       │
│    用 GPT-4/Claude 学习种子   │
│    批量生成 2000~3000 条数据   │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│ 3. 规则编译器（自动过滤）      │
│    Python 脚本检查 Markdown   │
│    语法和 Protégé 类名合法性   │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│ 4. 专家人肉（抽样精修）        │
│    修正关键业务逻辑错误        │
│    形成最终训练集              │
└──────────────────────────────┘
```

### 各步骤详解

#### 步骤 1：编写种子数据（Seed Data）

- 专家亲自动手编写约 **50 条**教科书级别、绝对正确的映射范例
- 这 50 条是整个数据集的"基因模板"，质量直接决定最终模型水平

#### 步骤 2：LLM 数据扩增（Self-Instruct）

- 将 50 条完美范例作为 Few-Shot 提示
- 喂给最强商业大模型（GPT-4o / Claude 3.5 Sonnet）
- 投喂海量原始寿险文本，让它模仿思维范式和 Markdown 格式
- 批量生成 **2000~3000 条**训练数据

#### 步骤 3：自动化语法校验（Compiler-Assisted Filtering）

- 写 Python 脚本检查批量生成的数据
- 检查项：
  - Markdown 括号是否配对
  - 引用的 Protégé 类名是否存在
  - 双链格式是否正确
- 不合法的数据直接滤掉

#### 步骤 4：人工抽样精修

- 对剩余高质量数据进行抽样检查
- 修正严重的保险业务逻辑错误
- 形成最终训练集

---

## 四、避坑指南："要与不要"

### 【不要】一味追求数量

- 垂直领域行为对齐：**1000 条高质量 CoT 数据 >> 10000 条格式混乱的保险问答**
- 数量太多小模型吃不下，反而导致格式混乱

### 【要】保持格式绝对纯净

- 每一条数据中，Markdown 层级标题（`#`、`##`）、双链格式（`[[...]]`）必须高度统一
- 模型微调时会极度敏锐地捕捉"统计规律"
- 教科书本身格式混乱 → 微调出来的 Markdown 就是灾难

### 【要】数据分布均衡

- A/B/C 三类题型比例大致协调（各 ~30%）
- 偏科的数据集会导致模型能力不完整

---

## 五、总结

准备微调数据集 = 为小基座模型编写一本极其严谨的**《保险本体与本地 Markdown 知识库操作期末考试真题及详解》**。

教材的逻辑有多清晰、格式有多规范，最终微调出来的"虚拟分析师"就有多好用。
