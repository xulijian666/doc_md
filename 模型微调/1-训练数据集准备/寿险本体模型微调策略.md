# 寿险本体模型微调策略分析

## 项目背景

基于 FIBO 本体和保险需规，构建寿险领域的需求分析模型。

---

## 三大范式适用性分析

### 1. SFT（有监督微调）— 必选、核心

**结论：必须使用，是项目的灵魂。**

**为什么不可或缺：**
- 核心任务：让模型理解寿险需求文档（需规），并将其映射到 FIBO 本体或数据表结构上
- 通用模型不具备这种"翻译"能力
- SFT 的角色是"逻辑和行为的塑造者"

**SFT 训练什么：**
精心准备"寿险业务文本 → 结构化本体映射"的问答对：

| 输入 | 输出 |
|------|------|
| 寿险需规中关于"定期寿险的观察期与身故给付责任"的描述 | 严格标准的 JSON 格式，准确包含：实体提取（如 `FIBO:InsuredParty`）、属性关联（如 `FIBO:hasPremium`）、映射规则 |

**核心价值：**
- 不是让模型背诵所有寿险条款
- 而是教会模型一种"思维范式"——输入任何寿险需求文档 → 条件反射般输出符合本体规范的分析结果
- 直接决定后续 D 步骤（工作量评估、基线检索）能不能跑通

---

### 2. CPT（继续预训练）— 备选，视情况而定

**结论：看情况使用。小基座模型太"笨"或私有知识太冷门时才需要。**

**什么时候需要用：**
- 小基座模型（如 7B）读寿险文档时表现像外行
- 连最基础的寿险专有名词都不理解：现金价值、保单贷款、复效期、减额交清
- 频繁理解错词义
- 直接做 SFT 效果会很差

**CPT 怎么做：**
把 A 步骤和 B 步骤涉及的所有静态资料灌进去：
- FIBO 本体官方全量概念解释文档
- 大量标准寿险条款、精算规定、保险法法律条文
- 现有企业内部保险数据表字典
- 过往的需规 MD 文档

**CPT 的目的：**
- 不是让模型学会回答问题
- 而是让模型先拥有满脑子的"寿险专业词汇量"和"行业常识背景"
- 有了这个底子，后面再做 SFT 事半功倍

> **评委视角提示：** 如果基座模型对中文保险理解尚可，跳过 CPT 直接做 SFT 就能出不错效果。但加上"CPT 注入寿险领域知识 → SFT 规范本体映射行为"的方案描述，技术架构看起来更专业饱满。

---

### 3. DPO（直接偏好优化）— 现阶段不建议

**结论：Demo 和比赛阶段暂时不需要。**

**为什么不需要：**
- **任务性质不同：** DPO 解决的是"开放式对话中的主观偏好"问题（礼貌性、偏见、幻觉）
- **你的任务是"硬性对齐"：** 对就是对，错就是错（实体有没有漏、JSON 格式对不对）
- **强逻辑、高确定性任务**通过 SFT 的高质量正向反馈已足够解决
- **成本性价比极低：** 收集 DPO 所需的"好回答 vs 坏回答"对照数据集极其耗费人力

---

## 推荐 Pipeline 概念蓝图

```
[海量寿险/FIBO原始文档]
        │
        ▼
   ┌─────────────────────────┐
   │  CPT: 补充寿险背景知识  │  ← 可选
   │  （让模型拥有寿险常识）  │
   └─────────────────────────┘
        │
        ▼
   [具备寿险常识的基座模型]
        │
        ▼
   ┌─────────────────────────┐
   │  SFT: 训练逻辑映射行为  │  ← 必选
   │  （需规文本→本体映射对） │
   └─────────────────────────┘
        │
        ▼
   [寿险需求分析专家模型]
```

## 实际落地建议

| 组合方案 | 适用场景 | 优先级 |
|----------|----------|--------|
| **直接 SFT** | 基座模型通用能力已足够，只需特定语气或格式 | ⭐⭐⭐ 首选 |
| **CPT → SFT** | 模型必须先吃透行业法条术语，再训练咨询解答 | ⭐⭐ 备选 |
| **SFT → DPO** | 已有专业能力，需防止说错话，让回答更简明 | ⭐ 暂不需要 |
