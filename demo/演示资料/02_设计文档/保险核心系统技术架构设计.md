## 1. 架构概述

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| 高可用 | 系统可用性 ≥ 99.95%，核心链路支持同城双活 |
| 高性能 | 核心交易接口 P99 ≤ 500ms，支撑日均 50 万笔投保 |
| 可扩展 | 微服务架构，按业务域独立扩缩容 |
| 安全合规 | 符合银保监会信息安全等级保护三级要求 |
| 开放集成 | 统一API网关，支持多渠道接入 |

### 1.2 技术选型

| 层次 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | Vue 3 + TypeScript | 管理端与客户H5 |
| 移动端 | Flutter | iOS/Android双端 |
| API网关 | Kong + 自研插件 | 统一接入、鉴权、限流 |
| 服务框架 | Spring Cloud Alibaba | 微服务治理 |
| 注册中心 | Nacos | 服务注册与配置中心 |
| 消息队列 | RocketMQ | 异步解耦、削峰填谷 |
| 缓存 | Redis Cluster | 热点数据缓存、分布式锁 |
| 数据库 | MySQL 8.0 + TiDB | OLTP + 大数据量场景 |
| 搜索引擎 | Elasticsearch | 保单检索、日志分析 |
| 文件存储 | MinIO + OSS | 电子保单、影像材料 |
| 容器化 | Kubernetes | 容器编排与弹性伸缩 |
| 监控 | Prometheus + Grafana | 指标监控与告警 |
| 链路追踪 | SkyWalking | 分布式链路追踪 |
| 日志 | ELK Stack | 日志采集与分析 |

## 5. 关键设计

### 5.1 分布式事务方案

投保流程涉及多个服务的数据一致性，采用 **Seata AT 模式 + 消息最终一致性** 组合方案：

```
投保提交
    │
    ├── 本地事务：创建投保单（insur-app）
    │
    ├── Seata全局事务：
    │     ├── 扣减库存（product-center）
    │     └── 锁定费率（rating-engine）
    │
    └── 异步消息：
          ├── 通知核保（underwriting）
          └── 记录流水（report-service）
```

### 5.2 缓存策略

| 数据类型 | 缓存方式 | TTL | 一致性保证 |
|----------|----------|-----|-----------|
| 产品信息 | 本地缓存 + Redis | 10分钟 | 发布时主动失效 |
| 费率表 | Redis | 1小时 | 配置变更时刷新 |
| 用户Session | Redis | 30分钟 | 登录时写入 |
| 保单状态 | Redis | 5分钟 | 状态变更时更新 |
| 核保规则 | 本地缓存 | 5分钟 | 规则发布时广播 |

### 5.3 限流降级策略

| 层次 | 策略 | 参数 |
|------|------|------|
| 网关层 | 令牌桶限流 | 单渠道 QPS ≤ 1000 |
| 服务层 | Sentinel 熔断 | 异常比例 > 50% 触发熔断 |
| 数据库 | 连接池限制 | 最大连接数 200/实例 |
| 消息队列 | 消费限速 | 单消费者 ≤ 500 TPS |

## 6. 安全架构

### 6.1 网络安全

- 生产环境与办公网物理隔离
- 通过堡垒机进行运维操作
- 数据库仅允许内网访问
- 所有对外接口强制HTTPS

### 6.2 应用安全

- JWT Token 认证，Token有效期2小时
- 敏感接口二次验证（短信/人脸）
- SQL注入/XSS防护（WAF + 代码层防护）
- API接口签名验证（防篡改）

### 6.3 数据安全

- 教感字段AES-256加密存储
- 数据库审计日志全量记录
- 数据脱敏展示（证件号、银行卡号）
- 数据备份策略：全量日备 + 增量实时备份
