## 1. 日常运维

### 1.1 每日巡检

| 巡检项 | 检查内容 | 工具 | 阈值 |
|--------|----------|------|------|
| 服务器状态 | CPU/内存/磁盘使用率 | Grafana | CPU<80%, 内存<85%, 磁盘<90% |
| 应用状态 | 实例健康状态 | K8s Dashboard | 所有Pod Running |
| 数据库状态 | 连接数/慢SQL/主从延迟 | MySQL Monitor | 连接<80%, 延迟<1s |
| 中间件状态 | Redis/MQ/ES集群状态 | 各管理控制台 | 集群健康 |
| 日志检查 | ERROR日志数量 | Kibana | 日增ERROR < 100 |
| 业务指标 | 投保成功率/支付成功率 | 业务监控 | 成功率>95% |

## 2. 告警处理

### 2.1 告警级别

| 级别 | 定义 | 响应时间 | 处理时限 | 通知方式 |
|------|------|----------|----------|----------|
| P0-致命 | 系统不可用 | 5分钟 | 30分钟 | 电话+短信+钉钉 |
| P1-严重 | 核心功能受损 | 15分钟 | 2小时 | 短信+钉钉 |
| P2-一般 | 非核心功能异常 | 30分钟 | 8小时 | 钉钉 |
| P3-提示 | 性能下降/预警 | 2小时 | 24小时 | 邮件 |

### 2.2 常见告警处理

#### P0: 服务不可用

```
1. 确认告警（1分钟内）
2. 检查K8s Pod状态
   ├── Pod CrashLoopBackOff → 查看日志 → 回滚/修复
   ├── Pod Pending → 检查资源配额 → 扩容节点
   └── Pod Running但不健康 → 检查应用日志 → 重启/回滚
3. 检查网络连通性
4. 检查依赖服务状态
5. 如5分钟内无法恢复 → 启动应急预案
```

#### P1: 数据库告警

```
1. 检查连接数
   ├── 连接池满 → 检查慢SQL → 杀死慢查询
   └── 连接泄漏 → 重启应用实例
2. 检查主从延迟
   ├── 延迟>1s → 检查从库负载 → 扩容
   └── 延迟>10s → 检查网络/磁盘IO
3. 检查磁盘空间
   └── 清理binlog/临时文件
```

#### P2: 缓存告警

```
1. Redis内存使用率>85%
   ├── 检查大Key → 拆分/压缩
   └── 清理过期数据
2. Redis连接数过高
   └── 检查连接泄漏
3. 缓存命中率下降
   └── 检查缓存策略/TTL设置
```

## 3. 数据库运维

### 3.1 备份策略

| 备份类型 | 频率 | 保留时间 | 工具 |
|----------|------|----------|------|
| 全量备份 | 每日 02:00 | 30天 | mysqldump + xtrabackup |
| 增量备份 | 实时 | 7天 | binlog |
| 异地备份 | 每日 | 90天 | OSS同步 |

### 3.2 恢复流程

```
确认恢复需求
    │
    ├── 误操作恢复 ──▶ binlog恢复（精确到秒）
    │
    ├── 数据损坏 ──▶ 全量备份+binlog恢复
    │
    └── 整库恢复 ──▶ 全量备份恢复
```

### 3.3 慢SQL治理

```sql
-- 查看慢SQL
SELECT * FROM mysql.slow_log 
WHERE start_time > NOW() - INTERVAL 1 HOUR
ORDER BY query_time DESC LIMIT 20;

-- 查看执行计划
EXPLAIN SELECT * FROM t_policy WHERE customer_id = 12345;

-- 常见优化手段
-- 1. 添加索引
ALTER TABLE t_policy ADD INDEX idx_customer_id (customer_id);

-- 2. 优化查询
-- 避免 SELECT *
-- 避免在WHERE条件中对索引列使用函数
-- 大分页使用游标分页
```

## 5. 应急预案

### 5.1 机房故障

```
1. 确认故障范围（机房A/机房B）
2. 启动灾备切换
   ├── DNS切换至备用机房
   ├── 数据库主从切换
   └── Redis Sentinel自动切换
3. 通知相关方
4. 持续监控备用机房状态
5. 故障恢复后切回主机房
```

### 5.2 数据库主库故障

```
1. MHA自动切换（30秒内）
2. 确认从库提升为主库
3. 应用连接池刷新
4. 数据一致性校验
5. 修复原主库，重新加入集群
```

### 5.3 勒索病毒防护

```
1. 立即隔离受感染服务器
2. 确认备份数据完整性
3. 使用干净备份恢复
4. 排查感染路径
5. 加强安全防护
```
