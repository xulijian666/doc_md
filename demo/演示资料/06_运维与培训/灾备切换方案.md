## 1. 灾备架构

### 1.1 双机房部署

| 机房 | 位置 | 角色 | 资源 |
|------|------|------|------|
| 机房A | 北京亦庄 | 主机房 | K8s集群20节点，MySQL主库 |
| 机房B | 北京顺义 | 备机房 | K8s集群20节点，MySQL从库 |
| 机房C | 上海 | 异地灾备 | 冷备环境（按需启动） |

### 1.2 数据同步

| 组件 | 同步方式 | RPO |
|------|----------|-----|
| MySQL | 半同步复制 | ≤ 1秒 |
| Redis | Sentinel + 异步复制 | ≤ 5秒 |
| 文件存储 | OSS跨区域复制 | ≤ 1分钟 |
| 消息队列 | RocketMQ多副本 | 0（同步复制） |

## 2. 切换场景

### 2.1 场景分类

| 场景 | 影响范围 | RTO | RPO | 切换方式 |
|------|----------|-----|-----|----------|
| 单实例故障 | 单服务 | ≤ 30秒 | 0 | K8s自动重启 |
| 数据库主库故障 | 数据层 | ≤ 1分钟 | ≤ 1秒 | MHA自动切换 |
| 机房级故障 | 全部服务 | ≤ 30分钟 | ≤ 1秒 | 手动切换机房 |
| 区域级灾难 | 全部 | ≤ 4小时 | ≤ 5分钟 | 异地灾备启动 |

### 2.2 触发条件

| 条件 | 判断方式 | 决策人 |
|------|----------|--------|
| 机房网络中断 | 监控告警+人工确认 | 值班运维 |
| 机房电力故障 | 基础设施告警 | 运维经理 |
| 核心服务不可用 | 业务监控告警 | 技术负责人 |
| 数据中心灾难 | 管理层通知 | CTO |

## 3. 切换流程

### 3.1 机房切换流程

```
确认切换需求（5分钟内）
    │
    ▼
通知相关方（发布群+电话）
    │
    ▼
执行切换（按顺序）
    │
    ├── Step 1: DNS切换（将流量指向机房B）
    │
    ├── Step 2: 数据库主从切换（机房B从库提升为主库）
    │
    ├── Step 3: Redis Sentinel切换
    │
    ├── Step 4: 应用服务验证（机房B各服务健康检查）
    │
    └── Step 5: 业务功能验证（核心流程冒烟测试）
    │
    ▼
切换完成通知
    │
    ▼
持续监控（切换后1小时高频监控）
```

### 3.2 切换命令

```bash
# DNS切换
# 修改DNS解析，将api.insurance.com指向机房B的SLB IP
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch file://dns-switch-to-b.json

# 数据库切换（MHA）
masterha_master_switch \
  --conf=/etc/mha/app1.cnf \
  --master_state=dead \
  --dead_master_host=10.0.1.100 \
  --new_master_host=10.0.2.100

# 验证切换
kubectl get pods -n production --context=cluster-b
curl -s https://api.insurance.com/health
```

## 4. 回切流程

### 4.1 回切前提

- 原主机房故障已排除
- 数据同步完成（主从延迟=0）
- 原主机房资源就绪

### 4.2 回切步骤

```
确认原主机房恢复
    │
    ▼
数据同步校验
    │
    ├── 将机房B的数据同步回机房A
    │
    └── 校验数据一致性
    │
    ▼
执行回切（与切换流程相同，方向相反）
    │
    ▼
业务验证
    │
    ▼
通知完成
```
