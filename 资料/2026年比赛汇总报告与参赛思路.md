> 想象一下,你是一家保险公司的IT部门负责人。每次有新产品(比如新的保险产品)要开发,都要写大量的需求文档、整理各种规则、分析历史数据......这个过程既繁琐又容易出错。

## 三、四大任务详解与开源工具推荐

### 任务A:本体构建--把保险知识"数字化"

#### 通俗解释
> **什么是本体?** 想象保险行业有一本"百科全书",里面记录了所有概念(如"投保人"、"被保人"、"保额")以及它们之间的关系(如"投保人购买保单"、"保单包含险种")。把这本百科全书变成计算机能理解的格式,就是"本体构建"。

#### 3.1 FIBO本体与保险数据表映射

**FIBO是什么?**
- FIBO(Financial Industry Business Ontology)是金融行业的标准本体
- 相当于金融行业的"通用语言",定义了金融概念的标准表示

**开源工具推荐:**

| 工具 | 用途 | 官网 |
|------|------|------|
| **Protégé** | 本体编辑器(可视化建模) | protege.stanford.edu |
| **Apache Jena** | RDF/OWL数据处理 | jena.apache.org |
| **TopBraid Composer** (免费版) | 本体开发 | topquadrant.com |
| **OOPS!** | 本体缺陷检测 | oops.linkeddata.es |

**Demo示例思路:**
```
1. 导入FIBO标准本体
2. 分析保险业务数据库表结构
3. 自动映射:数据库字段 → FIBO概念
   例如:policy_holder_name → fibo:Party.FullName
4. 生成映射报告,可视化展示
```

#### 3.2 数据表结构分析及本体映射

**开源工具推荐:**

| 工具 | 用途 |
|------|------|
| **SchemaCrawler** | 数据库结构扫描与分析 |
| **DBeaver** | 数据库管理与元数据提取 |
| **RDF4J** | RDF数据存储与查询 |
| **Ontop** | 关系数据库到本体的映射(OBDA) |

**关键参考论文:**
> Sutejo等(2024)提出了一种从ORM元数据和关系数据库中学习本体的方法,通过分析ORM定义文件构建抽象语法树(AST),然后提取OWL本体组件。这种方法可以显著提高本体提取的准确性。

#### 3.3 程序分析及本体映射

**目标:** 从现有代码中提取业务逻辑,映射到本体

### 任务B：MD文档管理——让文档“活”起来

#### 通俗解释
> 保险公司有大量需求文档（需求规格说明书、设计文档等），这些文档通常是Word格式。我们需要把这些文档中的知识提取出来，补充到本体中。

#### 核心工具选型：Obsidian

| 工具 | 用途 | 说明 |
|------|------|------|
| **Obsidian** | 知识文档管理 | **核心工具**，支持双向链接、知识图谱可视化 |
| **python-docx** | Word转Markdown | 将业务人员的Word文档转为MD格式 |
| **Flashtext** | 术语自动打标 | 自动识别文档中的本体术语并加双链 |
| **Neo4j** | 图数据库 | 存储知识图谱，支持复杂查询 |

**为什么选 Obsidian？**
- 🔗 **双向链接**：`[[投保人]]` 自动建立概念关联
- 🕸️ **知识图谱**：内置图谱视图，可视化展示概念关系
- 📝 **纯Markdown**：数据存储在本地，无厂商锁定
- 🔌 **插件丰富**：Dataview查询、Templater模板等

---

#### 3.5 需规文档内容抽取及本体映射

**核心问题：Protégé 定义的本体如何与 Obsidian 链接？**

```
┌─────────────────────────────────────────────────────────────┐
│                    衔接逻辑                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Protégé (OWL格式)                Obsidian (Markdown格式)   │
│  ┌─────────────────┐              ┌─────────────────┐       │
│  │ Class: 投保人   │              │ 投保人.md       │       │
│  │ Class: 保单     │  ──解析──→   │ 保单.md         │       │
│  │ Class: 理赔单   │              │ 理赔单.md       │
│  └─────────────────┘              └────────┬────────┘       │
│                                            │                │
│                                            │ 被引用         │
│                                            ↓                │
│                                   ┌─────────────────┐       │
│                                   │ 需求文档.md     │       │
│                                   │ "[[投保人]]需..."│       │
│                                   └─────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**简单理解：**
- Protégé 中的每个 Class → 生成一个 Obsidian 的 `.md` 文件
- 这些 `.md` 文件就是“词典条目”
- 需求文档用 `[[词典条目]]` 链接到这些条目
- Obsidian 自动建立关联网络

---

**核心流程：Word → Markdown → 自动打标 → 知识图谱**

```
① Protégé 定义本体，导出 insurance.owl
        ↓
② Python脚本解析OWL，为每个概念生成 .md 文件
   → 投保人.md、保单.md、理赔单.md...
   → 存放到 Obsidian 的 00-本体定义/ 目录
        ↓
③ 业务人员写 Word 需求文档
        ↓
④ convert.py 将 Word 转为 Markdown
        ↓
⑤ tag.py 扫描术语，自动加双链
   → "投保人需在犹豫期内退保"
   → "[[投保人]]需在[[犹豫期]]内[[退保]]"
        ↓
⑥ 复制到 Obsidian 知识库
        ↓
⑦ 知识图谱自动形成
```

**关键脚本：**

| 脚本 | 输入 | 输出 | 功能 |
|------|------|------|------|
| `owl2md.py` | insurance.owl | 00-本体定义/*.md | 将本体概念转为MD文件 |
| `convert.py` | Word文档 | output/*.md | Word转Markdown |
| `tag.py` | MD文件 + 术语清单 | tagged/*.md | 术语自动加双链 |

---

#### owl2md.py 脚本逻辑

```text
输入：insurance.owl（Protégé 导出）

处理：
1. 解析 OWL 文件，提取所有 Class
2. 对于每个 Class：
   a. 创建 {类名}.md 文件
   b. 写入 YAML Front Matter（类型、父类）
   c. 写入类的描述（rdfs:comment）
   d. 写入属性表格
   e. 写入关联概念的双向链接 [[xxx]]

输出：
- 00-本体定义/投保人.md
- 00-本体定义/保单.md
- 00-本体定义/理赔单.md
- ...
```

**生成的文件示例（投保人.md）：**

```yaml
---
concept_type: 实体
parent_concept: 保险参与方
source: protege
sync_date: 2026-05-22
---

# 投保人

## 定义
投保人是指与保险公司订立保险合同，并按照合同约定负有支付保险费义务的人。

## 属性
| 属性名 | 类型 | 说明 |
|--------|------|------|
| 姓名 | string | 投保人姓名 |
| 年龄 | integer | 需≥18岁 |
| 身份证号 | string | 18位身份证号 |

## 关联概念
- [[被保人]]：投保人可以为自己或他人投保
- [[保单]]：投保人通过保单与保险公司建立合同关系
- [[核保规则]]：投保人需要满足核保规则要求

> 🔄 本文件由 Protégé 本体同步生成
```

---

#### tag.py 脚本逻辑

```text
输入：
- 需求文档.md（convert.py 输出）
- ontology_terms.json（术语清单）

处理：
1. 读取术语清单（如：["犹豫期", "退保", "投保人", ...]）
2. 用 Flashtext 扫描 Markdown 文档
3. 遇到术语，替换为 [[术语]]
4. 输出新的 Markdown

输出：
- tagged/需求文档.md（术语已加双链）
```

**打标效果：**

| 打标前 | 打标后 |
|--------|--------|
| 投保人需要在犹豫期内决定是否退保 | [[投保人]]需要在[[犹豫期]]内决定是否[[退保]] |
| 保险公司应在30日内完成理赔 | [[保险公司]]应在30日内完成[[理赔]] |

---

#### 术语清单来源

术语清单 `ontology_terms.json` 有两个来源：

1. **从 Protégé 本体自动生成**
   - 提取所有 Class 名称
   - 提取所有 Property 名称
   - 输出为 JSON 数组

2. **手动补充**
   - 业务专家补充的常用术语
   - 历史文档中提取的高频词汇

  1. 识别实体:产品类型(防癌险)、目标人群(老年人)、保额(50万)

     - 需要考虑50万保额对应的费率计算规则

## 四、保险行业全生命周期示例

### 场景:一款"百万医疗险"从需求到上线的全流程

> 本章将以"百万医疗险"产品为例,完整展示如何从零构建本体系统,并在实际业务场景中应用。

---

### 4.1 第一步:本体构建(对应任务A)

#### 🎯 目标:用 Protégé 构建保险领域本体

#### 📍 工具选型:Protégé

**为什么选 Protégé?**
- 斯坦福大学开发的**标准本体编辑器**
- 支持 **OWL/RDF** 标准格式
- 可视化编辑类、属性、关系
- 支持推理机进行逻辑校验
- 导出格式可被其他工具读取

---

#### 📍 Step 1:准备基础工具

| 工具 | 用途 |
|------|------|
| **Protégé** | 本体可视化编辑器 |
| **Neo4j Community** | 图数据库,存储知识图谱 |
| **Apache Jena** | RDF/OWL数据处理 |
| **SchemaCrawler** | 数据库结构扫描 |

---

#### 📍 Step 2:在 Protégé 中创建本体

打开 Protégé,创建 `insurance.owl` 文件,定义以下内容:

**核心类(Class):**
- `保险产品`:百万医疗险、重疾险、意外险等
- `投保人`:购买保险的主体
- `被保人`:受保险保障的对象
- `保单`:保险合同载体
- `理赔单`:理赔申请记录

**对象属性(ObjectProperty):**
- `投保`:投保人 → 保单
- `承保`:保单 → 保险产品
- `保障`:保单 → 被保人
- `申请理赔`:被保人 → 理赔单

**数据属性(DatatypeProperty):**
- 保单:保额、保费、投保日期
- 投保人:姓名、年龄、身份证号

---

#### 📍 Step 3:数据库结构自动映射

使用 **SchemaCrawler** 扫描保险业务数据库:

1. 连接数据库,获取所有表结构
2. 自动识别表名 → 类名(如 `policy_table` → `保单`)
3. 自动识别字段 → 属性(如 `policy_id` → `保单编号`)
4. 生成映射配置文件

---

#### 📍 Step 4:导出本体

在 Protégé 中导出为两种格式:

| 格式 | 用途 |
|------|------|
| **OWL/RDF** | 标准本体格式,用于推理和交换 |
| **CSV/JSON** | 概念列表,用于生成 Obsidian 模板 |

### 4.3 第三步:模型微调(对应任务C)

#### 🎯 目标:训练"保险需求分析专家"AI模型

#### 📍 本地环境搭建

**Step 1:准备训练数据集**

收集和整理保险领域的问答对数据,格式如下:

| 字段 | 说明 | 示例 |
|------|------|------|
| instruction | 任务指令 | "分析以下保险需求的完整性" |
| input | 输入内容 | "开发一款百万医疗险,保额100万" |
| output | 期望输出 | "需要补充:免赔额、等待期、续保规则..." |

数据来源:
- 历史需求文档中的问答记录
- 业务专家整理的标准问答对
- 从本体中自动生成的训练样本

建议准备 **500条以上** 的高质量训练数据。

**Step 2:安装微调工具链**

使用 **LLaMA-Factory** 作为一站式微调框架:

1. **获取工具**:从 GitHub 克隆 LLaMA-Factory 仓库
2. **安装依赖**:安装 Python 依赖包
3. **下载基座模型**:推荐使用 Qwen2-7B-Instruct 或 ChatGLM-6B 作为基座模型
4. **准备GPU环境**:建议至少 1 张 RTX 3090(24GB显存)

**Step 3:配置微调参数**

采用 **LoRA(低秩适应)** 方法进行高效微调,主要参数:

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| lora_rank | 8 | LoRA秩,控制适配器复杂度 |
| lora_alpha | 16 | LoRA缩放因子 |
| num_epochs | 3 | 训练轮次 |
| batch_size | 4 | 批次大小 |
| learning_rate | 2e-4 | 学习率 |

**Step 4:执行微调并评估**

1. **启动训练**:使用 LLaMA-Factory 的训练脚本,指定数据集和参数
2. **监控训练过程**:观察 loss 曲线,确保模型正常收敛
3. **模型评估**:使用测试集评估模型效果
   - 需求完整性检查准确率目标:>90%
   - 实体识别F1-score目标:>85%
   - 规则提取准确率目标:>80%

## 七、Palantir Ontology 参考架构

### 7.1 什么是 Palantir Ontology?

> **通俗理解:** Palantir 是一家专注于大数据分析的公司,它的核心理念是构建组织的**"数字孪生"(Digital Twin)**--把现实世界中的实体(如保单、客户、理赔)及其关系映射到数字世界中,形成一个完整的语义层。

**这与我们的比赛目标高度契合!** 我们要做的就是为保险行业构建这样一个"数字孪生"。

### 7.2 Palantir Ontology 核心概念

Palantir 的 Ontology 由以下核心组件构成:

| 概念 | 保险行业示例 | 通俗解释 |
|------|-------------|----------|
| **Object 类型** | 保单、客户、险种、理赔单 | 定义一类事物,类似于"类"(Class) |
| **Object(对象)** | 保单#001、客户张三 | 具体的个体实例 |
| **属性(Properties)** | 保额、保费、投保日期 | 对象的特征 |
| **链接(Links)** | 客户→购买→保单 | 对象之间的关系 |
| **操作(Actions)** | 核保、理赔、退保 | 可以执行的动作 |
| **函数(Functions)** | 风险评分计算、费率计算 | 复杂的业务逻辑 |

### 7.3 Palantir 的数据集与本体映射

Palantir 提供了一个清晰的映射关系:

| 数据集概念 | Ontology 概念 | 保险示例 |
|-----------|--------------|----------|
| 数据集 | Object 类型 | 保单表 → 保单(Object Type) |
| 行 | Object | 一行数据 → 一张保单(Object) |
| 列 | 属性 | 字段名 → 属性名 |
| 字段值 | 属性值 | 具体值 → 属性值 |
| 合并(Join) | 链接类型 | 表关联 → Object关系 |

### 7.4 为什么需要 Ontology?(Palantir 的观点)

Palantir 总结了 Ontology 的 5 大优势:

#### 1️⃣ 规模化连接
> **保险场景:** 不同部门(精算、核保、理赔、客服)都有自己的数据视图,Ontology 提供统一的"单一事实来源",避免各部门"重复造轮子"。

#### 2️⃣ 可解释性
> **保险场景:** 业务人员不需要理解数据库表结构,只需要用业务术语(如"保单"、"投保人")就能查询和分析数据。

#### 3️⃣ 规模经济
> **保险场景:** 一次构建本体,多个应用复用。新的保险产品开发可以直接基于现有本体,减少重复开发。

#### 4️⃣ 决策捕获
> **保险场景:** 核保决策、理赔决策都可以记录为数据,形成持续改进的反馈循环。

#### 5️⃣ 支持 AI/ML
> **保险场景:** 风险评估模型、欺诈检测模型可以直接映射到本体概念,简化模型部署。

### 7.5 Palantir 应用程序参考

Palantir 提供了多个基于 Ontology 的应用程序,我们可以借鉴其设计理念:

| 应用程序 | 主要用途 | 我们的对标 |
|---------|---------|-----------|
| **Object Views** | 单个对象的360度视图 | 保单全景视图 |
| **Object Explorer** | 搜索与分析 | 需求知识检索 |
| **Workshop** | 低代码应用搭建 | 需求分析工作台 |
| **Map** | 地理空间分析 | 客户分布分析 |
| **Vertex** | 图分析与关系探索 | 业务关系图谱 |
| **Foundry Rules** | 规则引擎 | 业务规则管理 |

### 7.6 Foundry Rules(规则引擎)

Palantir 的规则引擎设计理念值得借鉴:

```
规则编写 → 规则提案 → 审批流程 → 规则部署 → 规则执行 → 监控评估
```

**保险应用场景:**
- 核保规则数字化
- 理赔规则自动化
- 合规规则监控
